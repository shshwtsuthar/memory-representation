# Agent Data Protocol ‚Äî Code Review Guidelines

When reviewing this repository, be strict about dataset correctness and reproducibility. The repository's durable contribution rules live in `AGENTS.md`; read that file before reviewing any PR that touches `datasets/`, `agents/`, `schema/`, `scripts/`, or validation tests.

## Required Checks for Dataset PRs

For every dataset addition or dataset-format change, verify that the PR follows all applicable guidelines in `AGENTS.md`. In particular, check that:

- Required files are present: `README.md`, `extract_raw.py`, `raw_to_standardized.py`, `schema_raw.py`, `sample_raw.json`, `sample_std.json`, and root-level `sample_sft.json`. `api.py` is additionally required whenever the dataset emits any `ApiAction`.
- Top-level dataset JSON files are limited to `sample_raw.json`, `sample_std.json`, `sample_sft.json`, and `generated_thoughts.json`. No `full_*.json`, temporary chunks, downloaded corpora, scratch JSON, or alternate sample files such as `sample_fixed.json`.
- Optional `sample_sft/` subdirectory may contain agent-specific copies named `sample_sft_$AGENT.json` (e.g. `sample_sft_openhands.json`, `sample_sft_sweagent.json`). These supplement ‚Äî not replace ‚Äî the root `sample_sft.json`, must be regenerable from `sample_std.json` via the corresponding agent's `std_to_sft.py`, and must cover the same trajectories/IDs as the root sample.
- Sample files are generated by committed scripts and are not hand-patched fixtures. Mentally (or actually) re-run the pipeline: `sample_raw.json` ‚Üí `raw_to_standardized.py` ‚Üí `sample_std.json` ‚Üí `agents/<agent>/std_to_sft.py` ‚Üí `sample_sft.json` should reproduce the committed JSON. If sample JSON changed but the corresponding generator bug was not fixed, flag it.
- `sample_raw.json`, `sample_std.json`, and `sample_sft.json` represent **the same records in the same order**, with matching IDs between standardized and SFT stages. This is a hard requirement, not a soft preference.
- Sample size is small but representative ‚Äî normally 3‚Äì5 trajectories ‚Äî and covers important edge cases (tool calls, command output, final answers, dataset-specific action types, failures/rewards/terminal states where applicable).
- Extraction, standardization, and SFT conversion are deterministic so future contributors can reproduce the samples (no unseeded `random.*`, time-dependent behavior, or nondeterministic dict ordering in outputs).
- `schema_raw.py` validates `sample_raw.json` and standardized trajectories validate against the ADP schema.
- Every `ApiAction.function` exists in the dataset's `api.py`, and every `kwargs` object satisfies that function's Python signature (including required parameters such as the `message` argument for `finish`). If the dataset emits `ApiAction` without an `api.py`, flag it.
- SFT messages containing `<function=`, `<function_calls>`, or `<invoke name=` use `"from": "function_call"` (not `gpt`, `human`, `assistant`, etc.).
- `TextObservation.source` uses only schema-supported values: `user`, `agent`, or `environment`. Reject invented values like `system`, `os`, or `assistant`.
- Raw trajectory semantics are preserved: repeated actions, consecutive tool calls, observations, failures, rewards, and terminal states are not silently dropped. Any filtering must be implemented in code AND explained/justified in the PR description.
- Dataset-local `std_to_sft.py`, duplicate API definitions, or schema changes are clearly justified. Prefer shared agent converters in `agents/` whenever possible.
- Large corpora, full generated files (`full_raw.json`, `full_std.json`, `full_sft.json`), temporary chunks, caches, screenshots, and scratch JSON are not committed.
- Pre-commit hygiene is maintained: formatting, imports, trailing whitespace, and EOF newlines. **Every JSON file must be valid JSON and end with a trailing `\n`** (this is one of the most common PR issues ‚Äî check it explicitly).

## Required Validation Tests

Verify that the standard dataset validation tests pass (run them or confirm CI ran them):

```bash
python -m pytest tests/test_dataset_structure.py -v
python -m pytest tests/test_raw_schemas.py -v -k "<dataset_name>"
python -m pytest tests/test_standardized_schemas.py -v -k "<dataset_name>"
python -m pytest tests/test_std_to_sft_conversion.py -v -k "<dataset_name>"
python -m pytest tests/test_datasets_from_parameter.py -v
```

If the PR does not show test output (or a passing CI run), request that the contributor paste the results before merge.

## PR Description Requirements

Dataset PR descriptions must include **all** of the following. If any is missing or vague, request changes:

- **Dataset source** (paper, repo, or HuggingFace URL).
- **License** of the source data.
- **Size and split** used (e.g. number of trajectories, which split(s)).
- **Files added or changed** in this PR.
- **Schema mapping summary** ‚Äî how raw roles/actions/observations map to ADP types (`MessageAction`, `CodeAction`, `ApiAction`, `TextObservation`, `WebObservation`).
- **Tests run** ‚Äî which validation tests were executed and their results.
- **Known limitations** of the dataset or conversion.
- **Design-decision catalog** for unclear implementation choices (see below).

### Design-decision catalog

Each entry should include:

1. the ambiguity or question,
2. the chosen approach,
3. a concrete example from the dataset or code, and
4. alternatives considered and why they were rejected.

Examples of decisions that should be cataloged include handling system prompts, failed trajectories, unavailable tools, missing final messages, screenshots, rewards, metadata, action parsing, and filtering criteria. Include this catalog even when the decision seems small.

## Review Style

- Prefer actionable, must-fix comments over broad suggestions.
- Distinguish correctness issues from optional style improvements.
- Do not accept a promise to fix dataset alignment later; the PR should follow current ADP conventions before merge.
- If this review is produced by an AI agent, include a short disclosure in the review body that the review was generated by an AI agent.
